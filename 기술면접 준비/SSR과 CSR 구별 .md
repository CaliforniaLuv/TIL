  차이: 렌더링되는 위치가 서로 다르다.
  
  SSR: 웹 페이지를 서버 측에서 렌더링하며, 렌더링 작업이 완료될 경우 클라이언트 측에 보낸다.
  웹 페이지에서 데이터가 필요할 경우, 서버는 데이터를 완전히 불러온 다음 렌더링 작업이 완료된 페이지에 변환한 후, 클라이언트 측에 보낸다.
  브라우저의 페이지가 이동될 떄마다 서버는 같은 작업을 반복하므로 서버 부하가 커진다.
       
  CSR: 웹 페이지를 클라이언트 측에서 렌더링하며, 클라이언트에서 요청을 보내면 서버는 웹 페이지의 골격과 JS를 클라이언트에 보낸다. 받은 요소를 브라우저 즉 클라이언트 상에서 완전 렌더링된다.
  웹 페이지에서 데이터가 필요할 경우, API가 이용된다. 렌더링하는데 필요한 데이터를 API요청으로 서버와 클라이언트 간 상호작용을 이룬다.
  브라우저의 페이지가 이동될 때마다 서버가 웹 페이지의 골격과 JS파일을 보내지 않고 브라우저가 다시 렌더링한다.
  리엑트, 뷰의 SPA 기법은 CSR 방식에 중점으로 맞춰있다.
  
  SSR 사용 경우: 
   * SEO(검색엔진)이 우선순위인 경우 사용,
   * 브라우저와 사용자 간의 상호작용이 적은 경우(기사, 논문)
   * 첫 화면 렌더링이 빠르게 필요할 경우
  
  CSR 사용 경우:
   * SEO가 필요하지 않을 경우
   * 브라우저와 사용자 간의 상호작용이 많을 경우, UX / UI 요소가 다양한 경우
   
   SEO 검색엔진이란?
   
   검색엔진에 사용되는 자동화크롤러이며 첫 HTML을 호출하여 보이는 값을 크롤링(데이터 추출)한다. CSR은 브라우저 상 첫 HTML은 빈 화면이므로 SEO가 필요없다.
